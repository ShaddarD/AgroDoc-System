# Refactor Skill

Ты работаешь с существующим проектом AgroDoc-System.

Стек проекта:

* Django 4.2 + DRF
* React 18 + TypeScript
* Ant Design v5
* PostgreSQL 15
* Docker Compose

Главная задача:
делать безопасный, практичный рефакторинг существующего кода без ненужного переписывания проекта с нуля.

## Основные принципы

* Не переписывай большие куски кода без явной причины.
* Предпочитай минимальные изменения с максимальной пользой.
* Сохраняй текущее поведение системы, если пользователь не просит изменить бизнес-логику.
* Не ломай API-контракты без явного предупреждения.
* Если есть несколько вариантов, сначала предлагай наименее рискованный.
* Сначала анализируй текущую реализацию, потом предлагай изменения.
* Если код уже приемлемый, не выдумывай лишний рефакторинг.

## Как проводить рефакторинг

Всегда начинай с короткого анализа:

1. Что в коде сейчас делает этот модуль или фрагмент
2. Какие есть реальные проблемы
3. Какие изменения нужны
4. Какие риски у изменений

После этого показывай:

1. Список проблем
2. Предлагаемое решение
3. Обновлённый код
4. Что проверить после изменений

## Что искать в backend

Проверяй:

* перегруженные views
* перегруженные serializers
* бизнес-логику, размазанную по views/serializers/models
* дублирование логики
* слабую валидацию
* N+1 запросы
* отсутствие select_related/prefetch_related
* неочевидные side effects
* плохую обработку ошибок
* неявные зависимости между apps
* слишком толстые service functions без разделения ответственности
* небезопасные места в auth/permissions/input handling

При рефакторинге backend:

* предпочитай thin views
* сложную логику выноси в services/selectors/helpers только если это реально упрощает код
* не выноси всё подряд в абстракции ради абстракций
* сохраняй стиль Django/DRF, не превращай проект в чужую архитектуру

## Что искать во frontend

Проверяй:

* слабую типизацию
* большие компоненты с перемешанными UI/API/state обязанностями
* дублирование логики
* лишние re-render
* плохую обработку loading/error/empty states
* неаккуратную работу с forms
* неочевидные зависимости между компонентами
* слишком тесную связку UI и API
* нарушения React hooks rules
* неправильное использование Ant Design

При рефакторинге frontend:

* сохраняй существующий UX, если не просят его менять
* разделяй UI, API и state там, где это реально улучшает читаемость
* усиливай типизацию без лишнего усложнения
* не тащи новые библиотеки без необходимости

## Ограничения проекта

Учитывай существующую структуру проекта.
Не предлагай полную смену:

* Django на FastAPI
* React на Next.js
* Ant Design на другой UI-kit
* Docker Compose на Kubernetes

Такие предложения допустимы только если пользователь просит именно архитектурную миграцию.

## Формат ответа

Отвечай в таком формате:

### Анализ

* что делает код
* где реальные проблемы

### Что улучшить

* 3–7 конкретных улучшений

### Код

* показывай изменённые участки
* полный файл показывай только если это действительно нужно

### Риски / что проверить

* что может сломаться
* какие сценарии нужно проверить вручную

## Правила качества

* Не давай абстрактных советов без кода.
* Не пиши "можно улучшить" без объяснения, зачем.
* Если изменение спорное, помечай это явно.
* Если рефакторинг влияет на frontend/backend контракт, предупреждай об этом отдельно.
* Если изменение затрагивает БД, указывай это явно.
* Если можно сделать маленький безопасный шаг вместо большого переписывания, выбирай маленький шаг.

## Специальное правило для этого проекта

Особенно внимательно проверяй:

* связку frontend API вызовов и реально существующих backend endpoint'ов
* модели и места, где поведение зависит от существующей схемы БД
* генерацию документов, скачивание файлов, права доступа и роли
* формы, таблицы, фильтры и статусы заявок
* места, где код "частично работает", но может ломаться на edge cases

## Что нельзя делать

* Нельзя переписывать всё приложение целиком.
* Нельзя убирать существующую функциональность без предупреждения.
* Нельзя предлагать "идеальную архитектуру" вместо практичного шага.
* Нельзя ломать совместимость молча.
* Нельзя заменять рабочий код на более "красивый", если пользы почти нет.
